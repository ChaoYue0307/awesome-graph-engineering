> [!IMPORTANT]
> ここでの **Graph Engineering（グラフ・エンジニアリング）**は AI エージェント・システムを対象とします。グラフデータベース、知識グラフ、グラフ ETL、グラフニューラルネットワークは対象外です。

## 暫定的な定義

**グラフ・エンジニアリングとは、グラフ構造を持つエージェント・システムについて、役割と実行時インスタンス、それらを結ぶ契約、共有する状態と成果物、そして集団としての振る舞いを判定する証拠を明示し、実行・観測・進化させることで、システム全体を制御・テスト・改善可能な工学対象にする実践です。**

グラフは装飾ではなく、実行を支える構造でなければなりません。宣言されたトポロジー、実際に生じた実行グラフ、またはグラフ生成方針が実行を実質的に制約し、バージョン管理・追跡・評価・意図的な変更が可能な程度に検査できる必要があります。

**Graph Engineering は新興の実践を表す暫定的な用語であり、確立した学術分野や業界標準ではありません。** この定義は、マルチエージェント・システム、分散システム、ワークフロー・オーケストレーション、言語エージェント・グラフの研究を統合しています。各主張の出典は[エビデンスマップ](../../DEFINITION.md)にまとめています。

## 最小の三条件

システムが中核スコープに入るには、次の三条件すべてが実行上の意味を持つ必要があります。

1. **境界づけられた複数のエージェント実体**：少なくとも二つの個別に説明責任を持つ実行時インスタンスがあり、それぞれが文脈、権限、目的または責任を所有し、境界内で次の行動を選べること。
2. **契約に基づく協調**：制御、文脈、状態、成果物または評価をどう受け渡し、いつ結果を受理・拒否・エスカレーションするかを、少なくとも一つの明示的な関係が定めること。
3. **実行を支え、検査可能なグラフ成果物**：トポロジー、実行グラフまたはグラフ生成方針が実行を実際に制約し、バージョン化・追跡・評価・意図的変更ができること。

## 中核スコープに含まれないもの

- 多数のツールを使う単一エージェントは、依然として一つのエージェント・ノードです。
- 通常の決定論的関数だけで構成された DAG はワークフロー工学です。エージェント・グラフの基盤にはなれますが、それ自体が自動的にグラフ・エンジニアリングになるわけではありません。
- 契約、権限境界、状態規則、エビデンスゲートのない「ペルソナのグループチャット」は条件を満たしません。
- 実行を制約しない静的な組織図や事後的な可視化だけでは不十分です。
- グラフデータベース、知識グラフ、GNN、GraphRAG は、エージェント組織の協調または実行を直接支える場合に限り対象になります。
- エージェント数が多いほど良いとは限りません。単一エージェントの方が安価で信頼できるなら、より小さなシステムを選びます。

## コア・プリミティブ

| プリミティブ | 中心となる設計上の問い |
| --- | --- |
| **エージェント・ノード** | この文脈、目的、能力、権限境界を誰が所有するか？ |
| **型付きエッジ** | 何が、どの条件・スキーマ・失敗規則で関係を横断するか？ |
| **組織グラフ** | どの再利用可能な役割が委任・協調・検証・エスカレーションできるか？ |
| **実行／作業グラフ** | 今回の仕事に必要な依存関係、分岐、インスタンス、成果物は何か？ |
| **エビデンスゲート** | どの外部証拠が、続行・拒否・人への移管を許可するか？ |
| **状態境界** | 何を共有・隔離・チェックポイント化し、誰の記録を正本とするか？ |
| **グラフ方針** | 誰がノードやエッジを生成・経路変更・書き換え・取消できるか？ |

組織グラフは「誰が何を実行・検証できるか」、実行／作業グラフは「この実行で次に何が必要か」を表します。これは相補的な分析視点であり、二つの物理グラフを実装することを強制するものではありません。
